Guide opérationnel pour le déploiement du template MOFU Hub — 4 workstreams, responsabilités RACI, actions concrètes et livrables attendus par composant.

À qui s'adresse ce document
Chef de marché (porteur du template pour son univers), Chef de projet digital, Studio, Webmerch, Marketplace, équipes Data et Dev. Chaque section indique qui est concerné et ce qu'on attend d'eux.
Ce que ce document couvre
4 workstreams d'activation : les données produit à enrichir, les contenus à produire ou reformater, les avis à structurer, les services à inventorier. Pour chaque WS : quoi, pourquoi, qui fait quoi (RACI), livrables.
Ce que ce document ne couvre pas
La conception technique de la page (wireframe séparé), les choix graphiques (mockup séparé). Les délais et sprints sont à définir avec le chef de projet selon le calendrier de déploiement par univers.

Gouvernance & RACI

Légende — 4 niveaux de responsabilité
RResponsible — réalise le travail. Peut être plusieurs personnes.
AAccountable — valide et assume la décision finale. Un seul par livrable.
CConsulted — est sollicité pour input ou validation partielle, en amont.
IInformed — est tenu informé de l'avancement et des décisions prises.
Règle de gouvernance : chaque livrable ne peut avoir qu'un seul A. En cas de doute sur la validation, l'A remonte au Chef de projet digital. Le Chef de marché est A sur tous les livrables qui engagent son univers (données, contenu, services).

Définition des rôles
RôlePérimètreCe qu'il engage sur ce projet
Chef de marché Propriétaire de l'univers produit (jardin, cuisine…) Valide les données produit, les règles de badge, les services disponibles et le contenu éditorial. C'est son univers — ses décisions engagent la qualité finale du template.
Équipe Data / PIM Gestion du catalogue et du référentiel produit Enrichit les champs manquants (prix de référence, badge, type d'installation, indice de protection). Source unique de vérité pour les données produit.
Équipe Dev / IT Développement et infrastructure Expose les APIs catalogue, avis et CMS. Assure le temps de réponse et la disponibilité temps réel du stock. Intègre le template dans le CMS cible.
Studio Production éditoriale — contenus, guides, blog, FAQ Produit les textes GEO, reformate les guides existants, enrichit la FAQ avec les normes techniques, tagge les articles blog par univers. Interlocuteur éditorial unique pour tous les univers.
Webmerch Merchandising web — prix, badges, services, mise en avant Définit les règles de badge produit, valide les prix de référence affichés, gère le catalogue de services disponibles par univers. Garant de la cohérence entre l'offre et l'affichage template.
Marketplace Gestion des vendeurs partenaires et de leur offre Alimente le composant S10 Marketplace du template avec les offres partenaires pertinentes par univers. Valide la pertinence des produits tiers exposés dans le hub.
Chef de projet digital Coordination transverse Pilote le planning par univers, arbitre les priorités inter-workstreams, valide la mise en production. Point de contact unique entre les équipes.
Direction Sponsor budget et priorité Valide le périmètre des univers à déployer et l'ordre de déploiement. N'intervient pas dans le détail opérationnel.

WS 1 — Flux Produit & Données

Bloquant
Quoi
Enrichir les données produit du catalogue pour que chaque composant dynamique du template s'affiche correctement : prix, stock, type, badge, installation.
Pourquoi
Sans ces données, le catalogue affiche des cartes incomplètes, le calculateur retourne de mauvais produits, les tableaux comparatifs sont creux. C'est le workstream bloquant pour tout le reste.
Qui
Responsable : Équipe Data / PIM
Décideur : Chef de marché
Intégration : Équipe Dev / IT
Comment
Enrichissement dans le PIM (outil de référence produit), puis exposition via l'API catalogue. Les règles de badge sont définies par le chef de marché et implémentées par l'équipe Data.
Composants du template alimentés par ce workstream
S06 Catalogue (grille produits) S07 Calculateur (résultats) S08 Tableaux comparatifs S10 Marketplace partenaires
Données requises — état & actions à mener
Donnée État actuel Action à mener — par qui Priorité
Identifiant unique (SKU)
Réf. interne produit		 Disponible Aucune action. Vérifier cohérence avec l'ID utilisé dans les APIs avis et services. P1
Nom produit normalisé
Format : Marque · Modèle · Spec principale		 Partiel Équipe Data : Normaliser le libellé sur 100 % des SKUs MOFU de l'univers. Le format doit être lisible et cohérent sur toutes les cards produit. P1
Type de produit
Taxonomie fermée propre à l'univers		 Partiel Chef de marché : Définir la liste fermée des types pour cet univers. Équipe Data : Mapper les catégories existantes sur cette taxonomie. P1
Prix actuel de vente
Prix affiché sur le site		 Disponible Aucune action. P1
Prix de référence fabricant
PVPC — affiché barré sur la card		 Absent Équipe Data (acheteur) : Collecter le PVPC fabricant pour chaque SKU. Source : fiche constructeur, contrat acheteur ou base GS1. Requis pour afficher le prix barré. P1
Statut de stock
En stock / Dernières pièces / Sur commande		 Partiel Équipe Dev : Exposer le statut en temps réel. Chef de marché : Valider le seuil "Dernières pièces" (ex. stock < 5 unités). P1
Badge produit
Éco / Design / Connecté / Bestseller		 Absent Chef de marché : Définir les règles de badge pour son univers (critères objectifs par badge). Équipe Data : Appliquer les règles dans le PIM. P1
Classe énergie
Obligatoire — réglementation UE		 Partiel Équipe Data : Vérifier présence sur 100 % des SKUs. Champ obligatoire légalement — alerter si manquant sur des produits déjà en ligne. P1
Type d'installation
Sans travaux / Facile / Pro recommandé		 Absent Chef de marché : Valider la règle de dérivation automatique selon le type produit. Équipe Data : Implémenter la règle dans le PIM. P2
Délai de livraison estimé
Affiché sous le prix sur la card		 Partiel Équipe Dev : Calculer dynamiquement ou mapper statiquement sur le statut stock (en stock → 48h, sur commande → sous 7 jours). P2
Note & nombre d'avis
Affichés sur la card et les tableaux		 Disponible Équipe Dev : S'assurer que ces champs sont synchronisés depuis le système d'avis (WS3) — pas de valeurs statiques dans le PIM. P2
Spécification principale
Variable par univers (puissance, largeur, capacité…)		 Partiel Chef de marché : Définir quelle est la "spécification principale" pour son univers. Équipe Data : Vérifier exhaustivité et format homogène sur 100 % des SKUs. P1
GET /api/v2/catalog?universe={slug}&limit=20&offset=0 // Filtres supportés : ?type= · ?badge= · ?price_max= · ?stock=ok|low { "products": [{ "sku": "REF-001", "name": "Marque Modèle [Spec principale]", "type": "[type-univers]", "spec_main": "[valeur + unité]", "price_current": 89.90, "price_reference": 109.00, "stock_status": "ok", // ok | low | order | unavailable "delivery_estimate":"48h", "rating": 4.5, "review_count": 234, "badge": "eco", // eco | design | connecte | best | null "install_type": "easy", // easy | pro | sans-travaux "url": "/univers/marque-modele" }] }
Contrainte performance : Le calculateur filtre les produits côté client — le feed catalogue doit être chargé en moins de 300 ms. Prévoir un cache CDN sur les réponses par univers.
Livrables attendus — WS1
1Taxonomie des types produit validée — liste fermée, mappée sur les catégories PIM existantesChef de marché + Data
2Règles de badge documentées et implémentées dans le PIM — critères objectifs par badge, par universChef de marché
3Prix de référence fabricant (PVPC) renseigné pour 100 % des SKUs de l'univers concernéData / Acheteur
4API catalogue exposée avec filtres type / badge / prix / stock — temps de réponse <300 ms, testé en chargeDev / IT
5Statut stock temps réel ou cache court (TTL <15 min) pour le badge "Dernières pièces"Dev / IT
Point d'attention : Sans les livrables 1 (taxonomie) et 3 (PVPC), les filtres du catalogue et le prix barré sont non fonctionnels — le template ne peut pas être mis en ligne. Ces deux points sont sous la responsabilité du Chef de marché.

WS 2 — Gestion Éditoriale

Prioritaire
Quoi
Identifier les contenus existants (guides, articles, FAQ), les reformater selon les contraintes du template, et produire les contenus manquants (intros GEO, normes techniques).
Pourquoi
Le template est conçu pour les moteurs de recherche IA (SGE, Perplexity) autant que pour Google. La qualité factuelle et le respect des formats GEO déterminent la visibilité organique.
Qui
Responsable : Studio
Décideur : Chef de marché
Validation : Chef de projet
Comment
Audit du corpus existant par univers, sélection et reformatage, puis production des contenus manquants selon les gabarits ci-dessous. Livraison dans le CMS ou en fichier structuré (JSON/CSV).
Composants du template alimentés par ce workstream
S03 Chapeau hero S08 Intros GEO tableaux S12 FAQ (6 Q&A) S13 Actualités (3 cards) S14 Conseils (5 cards)
Contenus à sourcer, reformater ou produire
ContenuSourceAction + GabaritPriorité
Intros GEO tableaux
2 textes × univers (produits + marques)		 À rédiger — n'existe pas dans ce format Exactement 100 mots. Minimum 2 chiffres factuels vérifiables. Mots-clés SEO naturellement intégrés. Zéro formule générique. Livraison : CMS ou fichier texte. P1
FAQ — 5 Q existantes
Q1 ouverte par défaut		 Pages catégorie · SAV · FAQ site existant Sélectionner 5 questions MOFU pertinentes. Question <90 caractères, réponse 60–180 mots. Q1 doit couvrir la recommandation principale de l'univers. P1
FAQ — Q6 normes techniques
Expertise de niche obligatoire		 À rédiger par un expert univers 1 question + réponse couvrant les normes techniques de l'univers (certifications, indices, classes). Valider avec un expert technique avant mise en ligne. P1
Actualités — 3 articles
Fraîcheur max : 6 mois		 Blog · Rubrique éditoriale Identifier 3 articles récents pertinents pour l'univers. Taguer dans le CMS avec l'univers slug. Format card : image 16/9 + catégorie + titre <65 car + date + URL. P1
Conseils — 5 guides
1 par thème obligatoire		 Blog · Guides d'achat · Fiches conseils 1 guide par thème : achat · installation · économies/performance · technique · design. Titre <55 car + description <90 car + URL. Livraison : CMS. P1
Chapeau hero
2 phrases, 25–40 mots		 À rédiger par univers Couvre 3 intentions (rénovation, économies, design/ambiance). 1 bénéfice fonctionnel + 1 bénéfice émotionnel. Pas de prix dans le hero. P2
Chips sémantiques
10 chips — SEO + filtre catalogue		 Analyse search interne + outil SEO Définir 10 chips depuis les requêtes les plus fréquentes sur cet univers. Chaque chip doit correspondre à un filtre catalogue existant. Valider avec l'équipe SEO. P2

WS 3 — Avis Clients (UGC)

Important
Quoi
Catégoriser les avis existants par univers et exposer un endpoint filtré pour alimenter la note globale dans le hero et les 3 avis sélectionnés dans le bloc UGC.
Pourquoi
La preuve sociale (note + avis) dans le hero augmente le taux de clic. Les avis dans le template sont des signaux UGC pour le SEO. Sans catégorisation par univers, il est impossible de filtrer les bons avis.
Qui
Responsable : Équipe Data / PIM
Décideur (API) : Équipe Dev / IT
Validation : Chef de projet
Comment
Ajout d'un champ "univers" dans le modèle avis (dérivé de la catégorie produit). Exposition de 2 endpoints : résumé pour le hero, liste filtrée pour le bloc UGC.
Données requises — état & actions
DonnéeÉtatAction — par quiPriorité
Champ "univers" sur les avis
Clé de filtrage par page hub		 Absent Équipe Data : Ajouter le champ univers en le dérivant de la catégorie produit associée à chaque avis. Sans ce champ, impossible de filtrer les avis par page hub. P1
Indicateur "vérifié"
Requis pour la note hero		 Disponible Aucune action. La note affichée dans le hero ne compte que les avis vérifiés. P1
Texte de l'avis
Bloc UGC — min. 20 mots		 Disponible Équipe Dev : Filtrer côté API les avis avec moins de 20 mots. Les avis trop courts dégradent la perception qualité du hub. P1
Auteur (prénom + initiale)
RGPD — jamais nom complet		 Disponible Vérifier que le format exposé est bien prénom + initiale (ex. "Marie D."). Ne jamais exposer un nom complet — exiger la conformité RGPD à l'API level. P2
Tag persona (optionnel)
Pour pages MOFU profondes uniquement		 Absent Studio : Tagger les avis les plus riches avec un profil intention (rénovation, design, budget). Réservé aux pages MOFU profondes, pas au hub. P3
// Résumé pour le hero : GET /api/v2/reviews/summary?universe={slug}&verified=true { "average_rating": 4.7, "total_verified": 1284 } // 3 avis pour le bloc UGC : GET /api/v2/reviews?universe={slug}&verified=true&min_rating=4&min_words=20&limit=3&sort=helpful { "reviews": [{ "rating": 5, "text": "...", "author_name": "Marie D.", "verified": true }] }
Règle éditoriale : Le hub n'affiche que des avis non segmentés. Les avis avec tag persona sont réservés aux pages MOFU profondes.

3Endpoint liste /reviews?universe= avec filtres verified, min_rating=4, min_words=20, sort=helpfulDev / IT
4Conformité RGPD vérifiée — prénom + initiale uniquement exposé dans les réponses APIDev / IT + Juridique

WS 4 — Services

Cadrage
Quoi
Recenser les services disponibles pour chaque univers (livraison, installation, financement, retrait), les structurer dans un modèle de données et les exposer via un endpoint.
Pourquoi
Les services affichés dans le template (S09 + card produit "Installation dès X€") doivent être exacts et disponibles pour l'univers. Afficher un service non disponible dégrade la confiance client.
Qui
Responsable : Chef de marché
Intégration : Équipe Dev / IT
Validation : Chef de projet
Comment
Le chef de marché audit les pages services existantes pour son univers et valide lesquels sont actifs. L'équipe Dev structure et expose ces données. 4 services affichés toujours — règle de fallback si <4 disponibles.
Disponibilité services — à vérifier par univers
ServiceDisponibilitéAction à menerPriorité
Livraison à domicile
Délai + conditions		 Universel Personnaliser le libellé avec le nom produit de l'univers. Vérifier le délai (48h ou 72h selon l'univers). P1
Service installation
Prix à afficher sur la card produit		 À vérifier Chef de marché : Confirmer disponibilité et prix de départ pour son univers. Ce prix apparaît sur chaque card produit du catalogue. P1
Click & Drive
Retrait en magasin		 Universel Vérifier si le délai est 2h ou 4h selon le format de produit (encombrant vs standard). P2
Financement
Avantage carte fidélité		 Universel Vérifier que l'offre est toujours active avant chaque déploiement d'univers. Contenu quasi-statique. P2
Reprise / Recyclage
Service optionnel — 5ème card		 À vérifier Si disponible pour l'univers, peut remplacer l'un des 4 services standard. Chef de marché : valider la pertinence. P3
Règle des 4 services : Le template affiche toujours exactement 4 services. Si un service n'est pas disponible pour un univers, il est remplacé par un service universel (Livraison, Click & Drive, Financement). Ne jamais afficher un service non opérationnel.
Livrables attendus — WS4
1Matrice services disponibles — tableau validé par le chef de marché pour chaque univers (disponible / non disponible / prix)Chef de marché
2Prix d'installation confirmé par univers — nécessaire pour l'affichage "Installation dès X€" sur chaque card produitChef de marché
3Endpoint /services?universe= retournant max 4 services actifs avec titre, description, prix et disponibilitéDev / IT
